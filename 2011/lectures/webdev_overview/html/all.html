Big Picture

Your Instructors

Brief Web History

  • Developed by Tim Berners-Lee et al. in 1989-1991
  • Conceived to share papers and experimental data for Physics
  • Every major OS included Web access software by 1997
  • Grew organically into the most popular application platform

Lots of Terms and Acronyms

  • DNS, FQDN, TLD, TCP/IP, ISP, Ethernet, LAN, 802.11, GSM, EDGE, 3G, EVDO, domain, IP address
  • HTML, CSS, JS, JSON, XML, HTTP, HTTPS, WAP, cookies, proxy, applets, plug-ins, extensions, Flash, Silverlight
  • ASP, JSP, PHP, RoR, API, SLA, EC2, S3, DC, rack, server
  • SQL, NoSQL, MySQL, Memcache, schema, tables, records, expiration policy

Everything Comes Down to This

  • One usage scenario.
  • One protocol.

Usage Scenario

  1. Publisher (author) loads data on a server.
  2. Consumer (user) points client software (browser) to the server.
  3. Client requests main document (page) from server.
  4. Client requests documents included by main document.
  5. Consumer follows (clicks on) a link in main document to reach another document.
  6. Rinse, repeat.

Big (Technical) Picture

  • Client gets a document from server.
  • HTTP (Hyper-Text Transfer Protocol)

HTTP Protocol

HTTP 1.0

1. Client opens TCP connection to server
1. Client sends HTTP request
1. Server sends HTTP response
1. Server closes TCP connection
1. Client closes TCP connection

HTTP 1.1

  • Optimization: use the same TCP connection for subsequent requests-response pairs
  • Still stateless: response determined by last request

Cookies

Concepts

  • Add state to HTTP, which is stateless
  • Each site gets a cookie jar
  • Cookie jar: key-value pairs, max size 4kb

Protocol

  • HTTP response header Set-Cookie sets cookies
  • Future HTTP requests have cookie jar in Cookie header

Find the cookie in the previous examples!

Server Side

  • a.k.a. The Back-End

10,000 Foot View

  1. Read HTTP request
  2. Write HTTP response

1,000 Foot View

  • Many server programs, possibly running on different machines.
    • Google has “hundreds of thousands” of servers.
    • Each Google query is processed by “hundreds” of machines.
  • HTTP request generally goes through the following.
    1. Load balancer
    2. Application server
    3. Database server

Load Balancer

  • Primary mission
    1. Receive HTTP request
    2. Relay request to one of many application servers
    3. Relay HTTP response to client
  • Overloaded with many tasks
    • Protocol management: security (HTTPS), compression
    • Security: clean up HTTP requests, prevent against DoS
    • Optimization: serve files in response to HTTP requests
  • Implementations
    • apache, haproxy, nginx

Application Server

  • Business logic
    • How does the homepage look like?
    • How do users sign in?
    • What happens when a user friends another user?
  • Talks to other servers (e.g. Facebook) via APIs.
  • Usually in high-level languages
    • PHP, Python, Ruby, Asp.Net
  • Each HTTP request is its own world
    • Use database to remember things between HTTP requests

Database Server

  • Stores data
    • Standard format: tables (like Excel spreadsheets)
    • Standard protocol: SQL
  • Solution to some very difficult problems
    • Concurrent requests (transactions)
    • Resiliency to failures (power outages, broken hard disks, etc.)
  • Implementations
    • Very hard to get right.
    • Used to be very expensive.
    • Free: MySQL, PostgreSQL, SQLite
    • Paid: Microsoft SQL Server, Oracle

Database Request Example
